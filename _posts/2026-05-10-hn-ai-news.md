  - title: LLMs corrupt your documents when you delegate
    link: https://arxiv.org/abs/2604.15597
    domain: arxiv.org
    summary: When you hand an LLM a document to edit, it quietly corrupts roughly 1-in-4 pieces of content — and the problem
      compounds across long workflows
    points: 380
    hn_url: https://news.ycombinator.com/item?id=48073246
    comments: 146
    time: May 9, 12:35 UTC
    content_bullets:
    - The authors introduce DELEGATE-52, a benchmark simulating long delegated editing workflows across 52 professional domains
      (coding, crystallography, music notation, and more) to test 19 different LLMs.
    - Frontier models corrupt an average of 25% of document content by the end of extended workflows — corruption worsens
      with file size, session length, and the presence of unrelated distractor files.
    - 'Errors are sparse but severe and accumulate silently: models make changes beyond what was asked, degrading documents
      in ways that may go unnoticed without careful review.'
    - Agentic tool use (giving the model external tools to act autonomously) did not improve fidelity — it failed to reduce
      the corruption rate on the DELEGATE-52 tasks.
    - Python code showed relatively stronger model performance than other formats, raising questions about whether training
      data composition drives format-specific reliability differences.
    discussion_bullets:
    - Commenters praised the round-trip evaluation method — running documents through chains of invertible transformations
      to catch accumulated errors — as an elegant way to measure corruption in an otherwise hard-to-quantify problem.
    - 'A recurring theme is ''semantic ablation'': each LLM pass strips away subtle meaning even when the surface text looks
      fine, and compounding across multiple sessions makes this worse.'
    - Practitioners sharing real-world experience noted that keeping documents small and single-purpose is the most reliable
      mitigation, and that LLMs are better used to write a deterministic script for conversions than to perform them directly.

    comments: 0
    time: May 9, 03:43 UTC
    content_bullets:
    - 'Root cause: SGLang''s scheduler was redundantly reopening the same CUDA IPC (GPU inter-process memory sharing) pool
      handles on every scheduler loop iteration, burning ~3% of total CPU time on throwaway bookkeeping.'
    - 'The fix: a Python dict caches pool handles after first open; since GPU memory pools are never reallocated, no cache
      invalidation logic is needed — only a lock on rare writes.'
    - 'Benchmarked on Qwen2.5-VL-3B-Instruct (a vision-language model) on H100 GPUs: throughput +16.2%, mean time-to-first-token
      -13.2%, end-to-end latency -10.6%.'
    - Decode latency improved even though the fix targets the prefill (input-processing) path — SGLang's single-threaded scheduler
      means any bottleneck slows everything downstream.
    - 'The patch (PR #21418) shipped in SGLang v0.5.10; profiling with py-spy showed 13% of scheduler CPU time in input processing
      before the fix.'
    discussion_bullets:
    - One commenter correctly identifies the core mechanism — caching tokenizer vocabulary lookups called millions of times
      — though the actual fix is more specifically about CUDA IPC pool handle reuse.
    - The thread briefly questions whether the optimization is architecture-specific; the fix targets SGLang's scheduler internals,
      so it should benefit any multimodal workload running on that framework.
    - A few comments use the post to praise Modal as an infrastructure platform, reflecting the broader theme that serving-stack
      tuning is increasingly a differentiator for ML deployment companies.
