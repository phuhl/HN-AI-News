    domain: rajveerbachkaniwala.com
    summary: Developer traces and fixes the three-layer clipboard mess that silently breaks image pasting in Claude Code on
      WSL
    points: 54
    hn_url: https://news.ycombinator.com/item?id=48267432
    comments: 84
    time: May 28, 02:28 UTC
    content_bullets:
    - WSLg's clipboard bridge converts Windows images to an old BMP variant (BI_BITFIELDS encoding) that Claude Code cannot
      read, causing silent failure with no error message.
    - Pushing a corrected PNG to the Linux clipboard triggers WSLg to sync it back to Windows and then overwrite it with the
      broken BMP format again, undoing the fix almost instantly.
    - Windows Terminal intercepts Ctrl+V before Claude Code can see it, so the image-paste keystroke never reaches Claude
      Code's handler.
    - 'The fix is three parts: a Windows exe (clip-listener.exe) that re-encodes clipboard images as proper PNGs, a Linux
      script that sets the PNG on the Linux clipboard and re-asserts it 0.5 s later to survive WSLg''s overwrite, and an Alt+V
      keybinding to bypass Terminal''s Ctrl+V interception.'
    - The full solution is open-sourced at github.com/rajveerb/wsl-clip-bridge and lets WSL users attach images in Claude
      Code by pressing Alt+V.
    discussion_bullets:
    - 'A commenter wryly noted the fix requires one more amendment: ''Or stop using Windows,'' sparking debate about whether
      WSL friction is a Windows problem or an acceptable tradeoff.'
    - Several replies agreed that pasting images into a terminal should be hard by design, reflecting broader disagreement
      about whether the clipboard bridge should even attempt seamless image passthrough.
    - Technical commenters highlighted that the three-layer translation between Windows clipboard, WSLg, and the terminal
      emulator makes this 'genuinely tricky engineering,' validating the depth of the investigation.
